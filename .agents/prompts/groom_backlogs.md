# Groom the Backlogs

Maintenance pass over `improvements.md`, `issues.md`, and `documentation/task_journals/`. Do not implement any backlog item during this pass.

**Subagents and delegation:** when re-verification fans out across many items or files, you may parallelize it with subagents or delegate it, subject to the delegation policy in `work_next_item.md`: read-only verification subagents (e.g. Claude Code's Explore agent) are allowed but bill the same plan as the session, so for limit-saving prefer headless Antigravity (`agy -p`) or local Ollama for self-contained verification briefs. All edits to the backlog documents, rankings, and journals stay in this session, and every claim a subagent or delegate reports must be spot-checked before it changes a row.

**Model fallback:** the models named in the backlog tables are suggestions, not requirements. Probe availability live before delegating and walk down this ladder until something works, skipping any rung whose tool is not installed or not logged in: (1) the item's recommended model for the host CLI you are running in; (2) other tiers on the same subscription (Antigravity: `agy models` lists live names, but expect the Gemini tiers (Flash and Pro) to share one account-wide quota — a quota error on one tier means the others are likely exhausted too, so step to GPT-OSS or another non-Gemini tier rather than another Gemini tier; Claude Code: step down Sonnet → Haiku); (3) local Ollama tags (`curl localhost:11434/api/tags`); (4) the session's own model doing the work inline. Only report the pass blocked if every rung fails — a groom is mostly reading and editing markdown, so any available model is acceptable.

1. Re-evaluate every Pending row in both documents: is it still worth doing, and do its requirements still match the current code and environment? Update, merge, or close stale items with a dated note; keep table rows and detail sections in sync.
2. Re-score and re-rank every Pending row using the diminishing-returns model defined in `improvements.md`: **Score = (Value 1–8 × Decay) ÷ Effort 1–8**, where Decay halves for each already-shipped item in the same theme (1.0 → 0.5 → 0.25 …) and stays 1.0 for new-capability items that open a new curve. Decay changes every time a same-theme item ships, so recompute all scores here rather than trusting last groom's numbers. Re-order Pending rows by score and keep the rationale column honest.
3. Apply the **ROI floor (0.5)**: flag every Pending row scoring below it with `⚠️ below floor` in its Status and a dated scoring note in its detail section. Below-floor items stay open — never close one unilaterally — but they must not be worked without explicit user confirmation, so surface each one to the user with a recommendation: confirm (work it anyway), re-scope (shrink it until it clears the floor), or close it. Clear the flag when a re-score lifts an item back above the floor.
4. Promote durable knowledge from recent sessions and agent memories into the backlog documents or `documentation/` so it survives outside any one agent's memory.
5. Journals: delete any journal whose item is no longer outstanding, and consolidate any journal too large to skim, preserving its Next Step line.
6. Commit the grooming as `docs(backlog): <summary>` and push.
